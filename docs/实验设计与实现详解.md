## 1. 用到的外部库

### 1.1 Skyfield（轨道动力学）

- **定位**：纯 Python 写的天文 / 卫星轨道计算库。在 `orbit/topology.py`、`orbit/skyfield_sim.py` 里调用。
- **本工作具体用法**：
  - `EarthSatellite(line1, line2, name, ts)` 从 TLE 字符串构造一颗卫星对象；
  - `sat.at(t)` 返回该卫星在时刻 `t` 的地心惯性坐标（km）；
  - 我们用这个三维位置算两星距离、星间链路是否被地球挡住、可见性等。
- **为什么用 Skyfield 而不是自己手写 SGP4**：Skyfield 内部就是包了 SGP4 + 时间转换；自己写容易出错（闰秒、TLE epoch、UT1 / TT 时间体系等坑）。
- **关键代码段（`orbit/topology.py`）**：把 0…3600 秒每个时隙都让 Skyfield 推算一次，存成形状 `[T, N]` 的张量缓存到 `.npz`，避免每次仿真都重算。

### 1.2 SGP4（轨道外推模型）

- **定位**：Simplified General Perturbations 4，业界标准的 LEO/GEO 卫星轨道外推算法。1980 年北美防空司令部公开。
- **本工作怎么用**：完全不直接调，藏在 Skyfield 内部。只要给一对 TLE 行，Skyfield 就用 SGP4 推算未来任意时刻的位置。
- **TLE（Two-Line Element Set）**：用两行 ASCII 描述一颗卫星轨道的标准格式，包含倾角、升交点赤经、偏心率、轨道周期等。本工作的 TLE 来源有两种：
  1. 从 CelesTrak（北美的卫星数据网站）下载真实 Starlink / Iridium TLE（需联网）；
  2. **回退方案**：`orbit/synth_constellation.py` 用开普勒第三定律算出周期，按照"等 RAAN 间距"原则**合成 16 颗 IPv6 网关 + 1 颗 AOS** 的 TLE。仿真器目前默认走这条路（`use_fallback_tle=True`），保证不依赖网络、严格可复现。

### 1.3 SimPy（离散事件驱动仿真）

- **定位**：Python 的轻量级离散事件仿真框架。在 `network/simpy_env.py` 里用作整个网络仿真的引擎。
- **核心概念**：
  - `env = simpy.Environment()`：一个事件队列 + 一个虚拟时钟；
  - `env.process(func())`：把一个生成器函数注册成"进程"。函数里用 `yield env.timeout(seconds)` 把控制权交回事件队列，等到那个虚拟时刻再继续；
  - `env.run(until=T)`：让事件队列一直跑，直到虚拟时钟到达 T。
- **本工作建了 3 类进程**：
  - `AOSSender`：每 1/300 秒（300 包/秒）产生一帧 AOS 报文，送给当前选定的网关；
  - `IPv6Gateway`：每颗网关一个进程，吃下入站报文、做协议转换、统计延迟；
  - `HandoffController`：每 1 秒读一次决策器输出，必要时触发两阶段迁移。
- **为什么用 SimPy 而不是写时间步进**：事件驱动天然支持「事件之间没动作就直接跳到下一个事件」，30 分钟仿真里 100 万事件，**单线程 3 秒就能跑完**；如果按毫秒步进遍历，要慢一两个数量级。

### 1.4 PyTorch

- **定位**：仅在 `optimizer/policy_net.py` 训练模仿学习小网络时用到。3 层 MLP，约 4.5 万参数，CPU/MPS/CUDA 任意设备都跑得动。
- **训练耗时**：MPS（M 系列 Mac）和 RTX 4090 上都是 30 秒左右。
- **`baselines/pure_drl.py` 里有 DQN 代码**，但当前的 4 方案版本不再使用（保留代码作为参考）。

### 1.5 其它辅助库

- `numpy` —— 数组运算、随机数、统计；
- `tqdm` —— 终端进度条；
- `matplotlib` —— 9 张图的全部绘制；
- `requests` —— 下载真实 TLE（默认走 fallback 时不联网）。

## 2. 实验总体步骤（4 方案版本）

`experiments/run_all.py` 跑完整 pipeline，分 5 段（原有 6 段，Stage 3 DQN 已删除）：

```
┌─ Stage 1：构建拓扑（约 5 秒）
│   合成 1 颗 AOS + 16 颗 IPv6 网关 → Skyfield 推算 0..3600 秒每秒位置
│   → 输出 TopologySnapshot 张量缓存
│
├─ Stage 2：生成 Lyapunov 专家轨迹 → 训练 IL 小网络（约 30 秒）
│   8 个训练负载场次 × 60 epoch
│   → 训练好的 il_policy.pt
│
├─ Stage 4：5 个测试负载场次 × 4 方案 = 20 次 SimPy 仿真（约 1.5 分钟）
│   每次仿真 1800 秒（30 分钟）× 300 包/秒 ≈ 54 万帧
│   → 每方案的 SimResults
│
├─ Stage 5：IL 推理延迟基准（约 10 秒）
│   CPU 上 2000 次推理，统计 mean / P99
│
└─ Stage 6：聚合 + 落盘（约 10 秒）
    → results/run_all_results.pkl + summary.json
    → plot_figures.py 出 9 张图
```

下面把每一步打开讲。

---

### Stage 1：构建拓扑

**做什么**

- 调 `orbit/synth_constellation.py` 生成 17 颗卫星的合成 TLE：
  - 1 颗 AOS（极轨，倾角 87°，高度 400 km）
  - 16 颗 IPv6 网关（倾角 53°，高度 550 km，等 RAAN 间距 22.5°）
- 调 `orbit/topology.py::compute_topology` 让 Skyfield 推算 0..3600 秒每隔 1 秒的：
  - 距离 `D[t, j]`（km）
  - 切线余隙角 `θ[t, j]`（不是地面仰角，是星间余隙角）
  - 可见性 `V[t, j]`（视线不被地球挡 + 距离 ≤ 3500 km）
  - ISL 带宽 `B[t, j]`（按 `isl_bandwidth_mbps` 距离线性插值 10–100 Mbps）
  - 传播延迟 `prop_delay_ms[t, j]`（距离 / 光速）
- 结果存成 `data/topo_*.npz`，下次跑同样配置直接加载，省 5 秒。

**为什么这样**

- 30 分钟仿真里平均 2–4 次切换才有意义。如果用真实 Starlink TLE，AOS 和网关都在 ≈ 53° 轨道面，几乎一直能看到同一颗，没法触发"切换"。合成星座**故意让 AOS 走极轨**，跟 53° 网关有显著相对运动，**每 5–10 分钟自然换星一次**。
- 1 秒采样足够（LEO 速度 7.6 km/s，1 秒位移 7.6 km，相对距离变化 ≪ 100 km，决策不会因为 1 秒分辨率出错）。

**难点 / 注意点**

- **切线余隙角不是仰角**：星间链路要算的是"两星连线最近点到地心的距离 > 地球半径 + 大气层厚度"，不是地面站的仰角。`orbit/skyfield_sim.py::has_line_of_sight` 用三维几何投影来判，避免常见错误。
- **TLE 校验**：合成 TLE 必须 checksum 正确，否则 Skyfield 解析失败。`_checksum` 函数手动算第 68 位校验码。

---

### Stage 2：生成 Lyapunov 专家轨迹 + 训练 IL 网络

这一步分两个子动作。

#### 2.1 生成专家轨迹（Lyapunov 在线决策器跑 8 遍）

- 调 `optimizer/lyapunov_solver.py::synthesize_gateway_loads` 生成 8 套不同负载（用种子 `42+0..7`）。每套是形状 `[3601, 16]` 的负载矩阵，每个元素在 [0, 1] 间表示某秒某网关的协议转换 CPU 负载。
- 对每套负载，调 `lyapunov_online(topo, loads)`：每秒读 80 维状态向量 → 算每颗候选网关的瞬时代价 → 加上虚拟队列 Q(t) 的切换押金 → 取最小者作为当前秒的动作 a*(t)。
- 一条 trajectory 包含 `states[3601, 80]` 与 `actions[3601]`，8 条拼起来共 ≈ 28 800 个 `(状态, 答案)` 样本。

#### 2.2 训练 IL 网络

- `optimizer/policy_net.py::GatewayPolicyNet` 是个 3 层 MLP：
  ```
  输入 80 → Linear → ReLU → Dropout(0.1)
       → Linear 128 → ReLU → Dropout
       → Linear 128 → ReLU → Dropout
       → 输出 16 维 logits
  ```
- 损失是 PyTorch 标准的 `nn.CrossEntropyLoss`（即 `-log π(a*|s)`，分类问题事实标准）。
- 优化器 AdamW，学习率 1e-3，batch 256，**60 epoch**，验证集占 20%。
- 训练日志写到 TensorBoard（`results/tensorboard/ours_il`）。
- 训练结束 `il_policy.pt` 存到 `results/checkpoints`。

**做完后的指标**：验证集动作匹配率 `val_acc ≈ 99.91%`，验证损失 `≈ 0.009`。

**为什么这样做**

- LEO 切换问题里"未来轨道完全可预测"，所以最优决策可以离线算出来；强化学习（DRL）那种"在线试错"是浪费——直接用 Lyapunov 这个"老师"标好答案，让小网络背一遍，更稳更快。
- 80 维状态向量 = 4 类特征 × 16 颗候选网关 + 一些 onehot 编码，整体保持"输入维度被问题尺寸完全决定，不需要试"。

**难点 / 注意点**

- **数据要足够多样**：只有 1 套负载，网络一定过拟合到这套负载的曲线。8 套独立场次保证网络在各种负载形态下都给出与老师相同的决策。
- **可见性硬约束要后置**：网络输出 16 个 logits，调 `predict()` 时显式把不可见网关的 logit 设为 −1e9，再取 argmax。这样无论网络怎么学，**绝不会选到当前不可见的网关**。

### Stage 4：5 个测试场次 × 4 方案 = 20 次仿真

这是实验的主战场。

#### 4 个对比方案

| 方案 | 决策怎么做 | 切换怎么做 | 代码 |
|---|---|---|---|
| **Reactive** | 当前网关不可见才换；选当前可见的、剩余可见时长最长的 | 硬切（直接断 + 重连，无状态迁移） | `baselines/reactive.py` |
| **Max-Visibility** | 每秒都选"当前可见的、剩余可见时长最长的"网关 | 硬切 | `baselines/max_visibility.py` |
| **MPTCP-style** | 综合分 = ΔT − 50 × L；当且仅当新候选综合分比当前**高出 60 秒**才切（迟滞机制） | 硬切 | `baselines/mptcp_style.py` |
| **Ours** | 加载 `il_policy.pt`，每秒前向推理输出网关编号 | 两阶段迁移（Pre-copy + Stop-and-copy + 三层降级） + Top-M 副本一致性 | `optimizer/policy_net.py` + `network/simpy_env.py::HandoffController._do_handoff` |

`Reactive` 和 `Max-Visibility` 是"弱基线"和"贪心基线"；`MPTCP-style` 是"强基线"（参考地面多路径思想），用 60 秒迟滞防乒乓抖动；`Ours` 是本工作。

#### 5 个测试负载场次

- 都由 `synthesize_gateway_loads(N, T, np.random.default_rng(seed))` 生成，**只有种子不同**：
  - 测试种子 = `42 + 999 + [0, 1, 2, 3, 4]` = `1041, 1042, 1043, 1044, 1045`；
  - 训练侧用的种子 = `42 + [0, ..., 7]` = `42, 43, ..., 49`；
  - **训练 / 测试种子完全不相交**，杜绝数据泄漏。
- 每"组"负载是一个 `[1801, 16]` 的矩阵 `L[t, j]`：第 j 颗网关在第 t 秒被其他业务占用了多少 CPU（[0, 1]）。生成规则：
  - 基础负载 ~ Uniform(0.1, 0.6)；
  - 加一个低频正弦扰动（周期 200–600 秒，振幅 ±0.2），代表其它业务的慢变；
  - 加一点高斯噪声（σ = 0.02）。
- 这 5 组场次差别就是"5 套不同的业务背景"，让算法在差异化场景里都跑一遍，平均掉运气。

#### 仿真本身怎么跑

进入 `network/simpy_env.py::run_simulation`：

1. 建 16 个 `IPv6Gateway` 进程，每个网关的基础负载 = 该种子下 `loads[0, j]`；
2. 建 1 个 `HandoffController` 进程，每 1 秒触发一次决策器，决策不同则触发 `_do_handoff`；
3. 建 1 个 `AOSSender` 进程，每 1/300 秒造一帧 AOS 报文，VCID 在 0..3 轮流，根据 `gateway_selector` 把帧塞给当前网关；
4. `env.run(until=1800)`：跑 30 分钟虚拟时间，约 100 万事件。

`_do_handoff` 分两条路径：
- **硬切**（Reactive/Max-Vis/MPTCP）：原网关 `is_active = False`，等 500 ms 物理切换窗口，新网关 `is_active = True`，原网关 dynamic 上下文整个销毁（所有半截分片全丢）。
- **两阶段**（Ours）：调 `migration/two_phase.py::precopy` → `stop_and_copy`，根据 `T_sync ≤ T_phys` 判定 SEAMLESS / DEGRADED / FAILED，再按结果搬迁 dynamic 上下文；Pre-copy 期间业务继续，只有 stop-copy 那一小段冻结。Top-M Gossip 副本逻辑同时在 `_gossip_loop` 里每 10 秒跑一轮。

**难点 / 注意点**

- **进程同步与传播延迟**：`AOSSender` 发包时调 `env.timeout(prop_ms / 1000)` 模拟单跳光速延迟，否则报文像"瞬移"，无法反映真实链路。
- **半截分片的会计**：硬切时 `partial_lost = frag_buffer.num_partial()` 是关键指标——这是"切换瞬间被销毁的、原本可能还能拼成完整 IPv6 报文的分片数"。
- **缓冲区溢出**：`in_buffer` 容量 1024，溢出的报文计 `drop_reason="buffer_overflow"`。MPTCP-style 频繁切换时偶尔会触发。

### Stage 5：IL 推理延迟基准

- `optimizer/policy_net.py::benchmark_inference_latency`（在 `run_all.py` 里调用）：
  - 先做 200 次预热（让 JIT、缓存、Python 解释器全暖起来）；
  - 然后跑 2000 次真实推理，记录每次墙钟时间；
  - 输出 mean / P50 / P99 / max。
- 当前机器（M 系列 MPS）实测 CPU 推理 0.075 ms / call；之前在 RTX 4090 上也是 0.14 ms / call 这个量级。
- 这一步存在的意义：证明本方案的决策步骤可以**装到星上**（决策周期 1 秒，推理 < 0.2 ms，富余 5000 倍）。

## 3. 性能指标定义与计算

所有指标都在 `network/simpy_env.py::SimResults` 里定义。下面逐项讲。

### 3.1 PLR（Packet Loss Rate，丢包率）

- **定义**：`dropped_frames / total_frames`。
- **怎么算**：每发一帧 `total_frames += 1`，掉的帧（无网关、网关 inactive、缓冲溢出）`dropped_frames += 1`。
- **为什么这么定**：业务层最关心的就是"我发了 100 个报文有几个没到"，这是端到端最直接指标。
- 4 方案对比里：Reactive/Max-Vis 都是 ~0.25%（硬切瞬间会丢一批），MPTCP-style 高达 6.4%（频繁切换叠加丢失），Ours 是 **0.000%**。

### 3.2 E2E（End-to-End Latency，端到端时延，ms）

- **定义**：`mean( delivered_at_sec − sent_at_sec )` × 1000，只算未掉的帧。
- **怎么算**：`AOSSender` 在发包时填 `sent_at_sec`；`IPv6Gateway` 处理完后填 `delivered_at_sec`；`SimResults.avg_e2e_latency_ms()` 算所有成功帧的均值。
- 4 方案对比：Reactive 6.07 ms（基线），Ours 6.24 ms（基线相当），MPTCP-style 8.60 ms（被切换打断更多）。

### 3.3 切换次数（num_switches）

- **定义**：30 分钟内 `_do_handoff` 被触发的总次数。
- **怎么算**：`len(SimResults.switches)`。
- 4 方案对比：Reactive/Max-Vis 都是 3.0（每次可见性结束被动切），MPTCP-style 69.6（迟滞机制在确定性场景反而触发频繁误判），Ours 1.0（决策器学会了"挑一颗能看全程的"）。

### 3.4 总中断时长（total_interrupt_sec, s）

- **定义**：所有 `SwitchRecord.interrupt_sec` 之和。
- 硬切：每次 `interrupt_sec = 0.5`（物理切换窗口）。
- 两阶段 SEAMLESS：`interrupt_sec = 0`。
- 两阶段 DEGRADED：`interrupt_sec = 0`（服务连续，仅部分包丢失）。
- 两阶段 FAILED：`interrupt_sec = physical_switch_sec`（最差等同硬切）。
- 4 方案对比：Reactive/Max-Vis 1.5 s（3 次切换 × 0.5 s），MPTCP-style 34.8 s（70 次切换），Ours **0.0 s**。

### 3.5 切换中丢分片数（fragments_dropped_in_switch）

- **定义**：所有 `SwitchRecord.fragments_dropped` 之和。
- 硬切时 = 原网关里"半截没拼完的"分片数；两阶段 SEAMLESS 时 = 0。
- 这个指标揭示了两阶段迁移的根本价值——**就算 PLR 是零，没把状态搬过去的话，本来已经拼到一半的分片全部白干**。
- 4 方案对比：Reactive 175 240、Max-Vis 145 279、MPTCP 146 853 ± 3 040、Ours **0**。

### 3.6 平均奖励（mean reward）

- 仅在 Lyapunov / DRL 训练阶段记录：每秒 `reward = -instant_cost`。聚合后得到一个标量。
- 在主表里不展示给评委（因为只对算法内部有意义）；在 IL 训练日志里能看到。

### 3.7 决策延迟（inference latency, ms）

- 由 Stage 5 单独基准测出来。本方案的小网络 CPU 0.075 ms / call。

### 3.8 负载均衡度 CV（Coefficient of Variation）

- **定义**：`std(usage) / mean(usage)`，其中 `usage` 是 `gateway_usage_count` 按 16 颗网关展开的列表。
- 越接近 0 越均衡。Ours CV = 0.64，MPTCP-style CV = 1.63。

### 3.9 Gossip 收敛指标（gossip_rounds, gossip_evictions, static_replicas_installed）

- 仅对启用了多副本一致性的本方案有效。每 10 秒 `_gossip_loop` 跑一轮，记录"广播了多少条 / 淘汰了多少条 / 安装了多少个副本"。
- 实测 30 分钟 ≈ 179 轮 Gossip，安装 1 副本、淘汰 1 个，验证三条路径（写 / 传播 / 淘汰）都打通。

## 5. 整体复盘 / 答辩可能被问到的点

| 可能的问题 | 回答要点 |
|---|---|
| "为什么用合成 TLE 而不是真实 Starlink？" | 真实 Starlink 卫星都在同一个 53° 倾角面，AOS 几乎一直能看到同一颗，30 分钟内没有真正的"切换需求"。合成星座让 AOS 走极轨、其他网关 53°，相对漂移快，触发自然切换 |
| "5 个测试种子代表什么？" | 5 套不同的业务负载场景，是 16 颗网关 30 分钟里的负载曲线；用 `mean ± std` 排除单次运气 |
| "决策周期 1 秒怎么定的？" | 必须 > 物理切换窗口 500 ms，必须 ≪ 可见窗口 3–10 分钟，与星上控制面 1 秒节拍对齐；1 秒落在交集 |
| "为什么不用 DRL？" | 卫星轨道完全可预测，Lyapunov 闭式就能给出最优；DRL 训练不稳、方差大、决策延迟波动大 |
| "为什么是 4 方案不是 6 方案？" | 早期版本含 Pure DRL（消融对比，已证明 reward 比 IL 差 4.7×）和 Ours-Lyap（理论变体，在线版的算法 1）。当前为了汇报聚焦，把 Lyap 当作"老师"留在原理章节、不参与对比；DRL 整组砍掉 |
| "fig6 显示 sync time 都远小于 500 ms 红线，那三层降级有意义吗？" | 当前仿真里链路条件正常，所有切换都走 SEAMLESS。三层降级是"链路异常时的兜底"，保证在最坏情况下也不会比传统硬切更差 |
| "如果重跑实验，数字会变吗？" | 不会。所有随机源都种子化（loads 用 numpy.default_rng，IL 训练 dataset split 用 torch.Generator.manual_seed），同种子结果完全相同 |
