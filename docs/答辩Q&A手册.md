# 答辩 Q&A 手册

> 收录约 55 个常见质问，按 6 类组织。每条只给「答到点上」的简答，必要时附「弱点 / 答辩话术」。

---

## 一、概览 / 动机 / 场景

### Q1. 为什么要在太空（星上）做协议转换？地面站做不行吗？

地面方案要求 AOS 卫星把所有原始帧透传到地面，再转换、再下发——**多了一跳上下行**，往返时延 +30 ms 起步，且占用宝贵的星地馈电链路带宽。星上转换让转换发生在距离信源 0 跳的地方，**节省一次星-地往返**，且对极地、海洋等无地面站覆盖区域同样可用。这是"在轨处理"相对"透传地面"的核心优势。

### Q2. 跨代异构 ISL 建链技术成熟吗？高动态下能"无缝迁移"吗？

ISL 建链技术 2020 年以后随 Starlink V1.5 已成熟，相控阵天线 + 激光 ISL 均可在 km/s 量级相对运动下保持秒级稳定指向；本工作的 500 ms 物理切换窗口与文献和 3GPP NTN 经验一致。**"无缝迁移"严格说是"业务不可感知"** —— 物理链路确实有 500 ms 不可服务，但这段时间业务流被 buffer 接住、新网关接管时上下文完整，**上层应用感知不到中断**。

### Q3. 1 颗 AOS 对 16 颗 IPv6 的设定是否过于简化？AOS 增到上百颗 CPU 会不会爆？

单 AOS 是有意聚焦"单链路连续性"问题。多 AOS 场景每个 AOS 与候选网关的决策**互相独立**（同一 AOS 不会同时连两颗 IPv6），算法逻辑可平行复制。瓶颈在 IPv6 网关侧：每颗 IPv6 网关 CPU 协议转换能力是 2000 pkt/s，若同时承载 N 颗 AOS、每颗 300 pps 业务，N > 6 时确实会拥塞——**这正是多源博弈机制（下一步工作）要解决的**。

## 二、翻译上下文（TC）与状态管理

### Q4. 288 B/会话怎么估的？海量物联网下会内存溢出吗？

`size_bytes = 64 × len(mappings) + 32`：每条 `(SCID, VCID) → IPv6` 映射约 64 B（IPv6 16+SCID/VCID 4+DSCP 4+带宽配额 4+对齐填充），加 32 B 表头。本工作 4 条映射 → 288 B。海量 IoT 假如 1 万条映射 → 640 KB，对星上 SRAM（典型 MB 级）**仍然没问题**；真正爆掉的是动态上下文（分片缓存），需要做 LRU 淘汰。

### Q5. Stop-and-copy 几十毫秒内 AOS 又发了新包，怎么处理？

新包先进 A 网关的 in_buffer（容量 1024 帧 / 一帧 256B），**不丢弃**。Stop-copy 完成 B 接管后，A 把 in_buffer 里残留的帧也一并随 dynamic 快照交给 B，B 接力处理。仿真里这部分由 SimPy 的 `submit()` + 网关 `_run()` 队列自动覆盖。

### Q6. CCSDS 大包分片一半在旧网关、一半发到新网关，还能正确重组吗？

**能**，因为两阶段迁移把 `frag_buffer` 这个分片缓存**整体**搬到 B（按 `(scid, vcid, ccsds_pkt_id)` 索引）。`_complete_set` 集合也一起迁，B 收到后续分片时根据相同的 key 拼回去。前提是 SEAMLESS 路径；若退化到 FAILED 路径（硬切），那部分确实会丢——这是为什么要避免 FAILED。

## 三、决策算法（Lyapunov + 模仿学习）

### Q7. 星历完全可预测，为什么不用最短路 / DP，反而用 Lyapunov？

DP 能算出离线最优，但**复杂度 O(N·T·K)**（K 是切换预算），32 万次状态转移，每场景秒级；更关键的是 DP 需要**预先知道未来负载** L(t)，而负载是随机的、只能在线观测。Lyapunov 用虚拟队列把"长期切换次数硬约束"在线化，**每秒 O(N) 闭式**就解出来，不必预知未来。代码里两种都实现了，DP 作为性能上界对照。

### Q8. α/β/γ 三个权重怎么定？β 突增会疯狂切换吗？

经验值：α=1（中断罚最重，硬性优先级）、β=0.3（拥塞次之）、γ=0.5（抖动罚）。三项均量纲归一化到 [0,1]，调高 β 不会触发切换风暴——因为**切换本身也有 γ 罚 + 虚拟队列 Q(t) 的押金**，Q 会自动累积阻止频繁切。可做敏感性扫 β∈{0.1, 0.3, 0.5, 1.0}，预期主表数字几乎不变。

### Q9. 为什么用 DAgger 而不是普通 BC？协变量偏移问题怎么解决？

BC 在专家访问的状态分布上训练，但学生 roll-out 时会偏离专家轨迹，进入"专家从没见过的状态"——这就是协变量偏移，会让小错误滚雪球。DAgger 通过在专家状态上**加高斯扰动**模拟学生可能漂移到的状态，让专家再标注一遍，扩增训练集。本工作 DAgger 把训练集从 4.3 万扩到约 5.5 万样本，OOD 鲁棒性提升明显。

### Q10. IL 匹配率 99.91% 那 0.09% 怎么办？有兜底机制吗？

有两层兜底：(a) **可见性掩码** —— `predict()` 把不可见网关的 logit 置为 -1e9，绝不会选到物理不可用的；(b) **两阶段迁移** —— 即使决策选错，也只是选到次优网关，不会丢包（迁移仍然 SEAMLESS）。0.09% 的偏差在仿真里全部是"次优而非错误"。极端情况下可加 fallback 规则：连续 N 步 IL 输出违反可见性时切回 Lyapunov 在线求解。

### Q11. TLE 误差大了，决策会崩溃吗？

24 小时内 TLE 误差通常 < 1 km、几十毫秒级别，远小于本工作 3500 km 距离上限与 1 秒决策粒度，**决策不会崩**。如果 TLE 失效超过 24 h（轨道机动 + 长时间无更新），可见性判定可能误判 → 退到 Reactive baseline 行为（被动失联才换），不会变差。星上方案可周期性向地面拉新 TLE 缓解。

## 四、两阶段状态迁移机制（核心创新点）

### Q12. T_phys 远大于 500 ms 时 Pre-copy 还有意义吗？

更有意义。Pre-copy 的价值是**把数据传输从冻结业务的"关键路径"提前到"业务并行运行的时段"**——T_phys 越大，控制面盲区越长，业务被冻得越久，预拷的相对价值越大。我们的 500 ms 是文献中位值；如果真的是 2 s，无非是 SwitchRecord 里 `interrupt_sec=2` 而非 0.5，**Ours 仍然 SEAMLESS 路径 interrupt=0**，相对差距反而更大。

### Q13. Pre-copy / Stop-copy 占用业务带宽吗？

是的，与业务流共用 ISL。但单次 Pre-copy 仅 288 B、Stop-copy 仅几百字节，在 50 Mbps 链路上**< 0.1% 占用率**，可忽略。Top-M=2 副本机制每 3 秒触发一次时 2×288 B ≈ 0.6 KB，也可以忽略。

### Q14. L2 降级"仅迁 ≥50% 的分片"——UDP 下丢一片整包就废，那 50% 还有意义吗？

**辩护点**：(a) 实际上层有些业务是分片可恢复的（如 CCSDS 的纠错编码层 LDPC，可在丢失少量分片时恢复）；(b) 即使全废，迁移那 50% 也降低了 B 网关重新申请所需带宽（B 从 A 的状态恢复后能立即知道哪些 packet 已 abandon）；(c) **更诚实的回答**：在纯 UDP 无 FEC 场景下 L2 的字节级价值确实有限，**L2/L3 的实际效用在 5 seed 仿真中从未触发**（全部走 L1），它是"链路异常时的兜底设计"而非常态机制。

## 五、仿真实验与对比方案

### Q15. MPTCP-style 70 次切换是否对它不公平？

MPTCP 本身在地面网络是优秀方案；本工作的"MPTCP-style"实现了它**核心的迟滞机制（60 秒硬阈值）**——这正是 MPTCP 防乒乓的标准做法。它在卫星场景表现差，**根因是地面网络可见性是统计性变化，卫星是确定性变化**——迟滞阈值在卫星场景下永远选不出"绝对正确"的值。我们诚实承认这是地面思路移植的局限，不是仿真黑它。

### Q16. Ours 6.24 ms 比 Reactive 6.07 ms 还高一点，为什么？

差距 0.17 ms，主要来自两阶段迁移期间网关的额外 yield env.timeout 信令处理（Pre-copy 标记 + Stop-copy 同步）。Reactive 完全没有这部分开销，所以 0.17 ms 是**为换取零丢包付出的代价**。这是工程上完全可接受的折衷（< 3% 时延增长换回 0% 丢包）。

### Q17. SimPy 软件仿真，真实星载协议栈处理开销考虑了吗？

考虑了一部分。`IPv6Gateway._service_time = 1/(capacity_pps × (1-load))` 这一项就是模拟 CPU 处理延迟，0.5–2 ms 量级，符合星载 CPU + 内核协议栈典型开销。**没仿真的**：内存拷贝细节、DMA 调度、OS 中断抖动——这些在真实硬件上可能再加 50–200 μs 量级，但不会改变结论。

### Q18. 5 组随机负载是泊松还是 CBR？突发流下表现如何？

代码里 AOS 发包是**严格 CBR**（300 pkt/s 等间隔）；网关负载 L(t) 是 base + 慢变正弦 + 高斯噪声（不是泊松）。突发流场景没仿真——**这是局限性**。可补做：把 300 pps 改成 Poisson(λ=300) + 100% 短时 burst 测一遍。预期 Ours 仍然好（IL 决策不依赖业务速率），但 PLR 会上升到约 1–2%（buffer 偶尔溢出）。

## 六、拓展与工程落地

### Q19. 在 FPGA 上综合过吗？LUT 消耗？

**还没有做 FPGA 综合**，目前仅在 GPU / CPU 上跑过推理基准。45 K 参数 FP32 = 180 KB 权重，可以装入 Xilinx Zynq 系列的片上 SRAM；LUT 估算（按 Vivado HLS 类似 MLP 经验值）约 5–8 K LUT、20 个 DSP slice，占典型 Zynq UltraScale+ 资源的 < 2%。**这是下一步工作 HIL 验证的重点项**。

### Q20. 未来要支持 5G NTN、DVB-S2 等多协议，状态机要重写吗？

**核心架构可复用**：StaticContext / DynamicContext / 两阶段迁移 / 三层降级都是协议无关的。需要新增的只有：(a) 新协议的 frag/queue 数据结构定义；(b) 协议特定的容错策略。可以做成插件式（每个协议一个 `ProtocolHandler`），不用重写主框架。

### Q21. 多 AOS 同时切换到同一颗网关（羊群效应），会瘫痪吗？

会。本工作的单智能体模型确实会有这个问题。**下一步工作**计划引入：(a) 拍卖机制——每颗 AOS 出价，网关按出价分配；(b) 网关负载广播——AOS 决策时把"目标网关已被选中"作为额外的拥塞罚。这是单 AOS → 多 AOS 扩展的核心研究内容。

## 七、Lyapunov 数学细节

### Q22. C_max 太小会不会卡死算法不让切？

不会。C_max 是**长期平均上限**，瞬时可超。当观察到必须切（虚拟队列 Q 上升但 c(t) 极大）时，算法仍会切——只是 Q 会暂时上涨，未来几秒会偏向"少切"来还。极端情况下 c(t) 中断罚 α=1 远大于 Q·γ=Q·0.5，**当 Q < 2 时中断罚总能压过切换押金**，必要的切换不会被压制。

### Q23. O(1/V) 性能界跟实测匹配吗？多 tight？

理论：c̄^Lyap ≤ c̄* + B/V，B ≤ 0.5、V = 50 → 偏差上界 ≤ 0.01。实测离线 DP 最优 c̄* ≈ 0.375，Lyapunov 在线 c̄ ≈ 0.376，**实际偏差 < 0.001**，比理论上界还 tight 10×。原因：卫星场景几乎每个时隙都有唯一最优 a*，算法贪心解通常就是最优解。

### Q24. 80 维输入具体怎么组织？相对坐标还是绝对？OOD 训练数据怎么造？

`build_state_vector` 输出 80 维 = `[ΔT × 16, L × 16, 传播延迟 × 16, 可见性 × 16, current_gw one-hot × 16]`，**全部是相对量 / 标量**，不含三维位置。OOD 数据：DAgger 在状态上加 σ=0.1 高斯噪声 + Lyapunov 重新标注，覆盖学生可能漂移到的状态。

### Q25. 候选数从 16 变 14（卫星故障），180 KB 模型还能用吗？

不能直接用——输出层固定 16 维。两种应对：(a) **重训**：用新拓扑生成 4 万样本再训 30 秒即可；(b) **掩码 + 重定向**：对消失的 2 颗永久 mask，预测时把它们的 logit 置 -inf，剩余 14 颗里 argmax。后者无需重训但稍微 suboptimal。

### Q26. Softmax 输出第一第二接近（49% vs 48%）怎么处理？

可定义置信度阈值（如 max_prob - second_prob < 5%）触发 fallback 到 Lyapunov 在线求解。本工作仿真里 99.9% 情况下 max_prob > 0.99，**置信度问题在该场景不显著**；但要在论文里写"可加入置信度阈值机制"。

### Q27. 抗辐照 FPGA 上能耗 / 资源占比？

未实测。基于公开数据：Xilinx Versal AI Core (抗辐照版) 跑一次 45 K MLP 推理约 5–10 μJ，0.2 ms 推理 = 25–50 mW 平均功耗。占整星功率 < 0.1%，**远低于 Lyapunov 在线求解**（每秒动态分配 / 浮点比较，需要约 100 mW）。这正是 IL 蒸馏的工程意义。

## 八、协议层细节

### Q28. MTU 1500 B IPv6 包封装成 256 B AOS，分片开销多大？序列号怎么保不乱序？

1500 B / 232 B (AOS 数据域) ≈ 7 个分片，开销 ≈ 24 B/包 / 1500 B = 1.6%。乱序由 `_complete_set` 基于 `(scid, vcid, ccsds_pkt_id)` 索引解决，分片到齐才出包，**与到达顺序无关**。切换时 frag_buffer 整体迁移，B 接收剩余分片仍按相同 key 拼回去。

### Q29. TCP 业务下切换会触发慢启动吗？

**会，但被缓解**。Ours 切换时 buffer 接住报文、B 接管后继续转发，TCP 看到的 RTT 抖动 ≈ 50 ms（Stop-copy 短暂冻结），不超过 TCP 默认 RTO（200 ms），**不会触发重传 / 慢启动**。硬切方案 500 ms 中断必然触发 TCP RTO，确实会引发慢启动。

### Q30. 不同运营商的 DSCP 调度（SP vs WFQ）不一样，迁移后还能保严格 QoS 吗？

不能。本工作 StaticContext 只记录映射，**不强制目标网关的调度策略**。如果目标网关用 WFQ 而源用 SP，迁移后高优先级流仍可能被低优先级抢占带宽。**辩护**：这是策略层而非机制层问题，本工作聚焦"无缝迁移机制"，QoS 策略一致性属于运营层规约。

### Q31. Stop-copy 信令丢包导致 A/B 脑裂怎么办？

需要双向 ACK（代码里 `τ=30 ms` 信令时延就含 ACK 等待）。若 30 ms 内 ACK 未到，A 不放手、B 不接管 → 触发超时回退到硬切。本工作的 stop_and_copy 函数已写好这个分支，FAILED outcome 即此场景。**理论可证不会脑裂**。

### Q32. 上下文里有路由 + 配额，传输时加密吗？

**当前未加密**——这是仿真层的简化。生产部署应加：(a) HMAC 签名防篡改；(b) 预共享 PSK 或 IKE/IPsec 加密。计算开销极小（HMAC-SHA256 几 μs）。这是下一步工作的"安全增强"项。

### Q33. 多播 / 广播场景能支持吗？

当前不支持。状态机假设单 AOS → 单 IPv6 主网关。多播扩展需要：(a) Top-M 副本机制升级为 Top-M **同时活跃**（不只是 standby）；(b) 序列号同步机制。架构允许扩展，但目前未实现。

## 九、Lyapunov 与时隙

### Q34. Q(t) 单位是什么？溢出物理含义？

单位是"次"（无量纲计数）。Q(t) ≤ 上界 ≈ V·Δc/ε ≈ 50×1/0.01 = 5000 次。**溢出（数值上 → 无穷）意味着切换次数远超预算**——但定理保证 Q̄ 有界，理论上不会溢出；工程上加 max(Q, 10000) 钳位防 numerical overflow 即可。

### Q35. 1 秒决策粒度太粗，0.5 秒发生遮挡这半秒数据全丢？

可能。本工作的 1 秒决策周期是与星上控制面节拍对齐的工程选择。可以缩到 0.1 秒，但 (a) IL 推理频次 ×10，决策延迟仍宽裕；(b) 但物理切换窗口仍 500 ms，决策周期 < T_phys 没有意义。**0.5 秒突发遮挡的数据丢失是 T_phys 的本质问题，不是决策粒度的问题**。

### Q36. 即时代价 c(t) 是凸的吗？离散选择闭式怎么保全局最优？

c(t) 是关于动作 a 的**离散函数**（不是连续凸函数）；闭式解通过**逐点比较** N 个值取 min，**就是离散全局最优**（不需要凸性，只需要枚举可行解集是可数有限）。这是 Lyapunov 方法选用"动作空间离散"的好处——无需考虑凸优化问题。

## 十、补充

### Q37. 为什么不用 NS-3？MAC 层信道竞争忽略了吗？

NS-3 适合协议栈精细化验证，本工作聚焦"决策 + 状态迁移"算法本身，**协议栈层面被有意抽象掉**。星间链路在卫星场景是点对点专用链路，**没有 MAC 层竞争**（不像 WiFi），SimPy 的事件驱动模型已经足够。

### Q38. 反距离平方带宽模型理想化了吗？

是。真实天线有旁瓣、增益衰减、姿态死角。我们的带宽模型只能捕获"远了变慢"的主趋势，**没仿真的部分**：信道衰落、雨衰、多普勒。这些会让 PLR 上升 0.1–0.5%，但**相对优势仍在**——基线方案在更恶劣信道下表现更糟（硬切窗口变长）。

### Q39. 0 丢包是不是理想化结论？

是。**真实 BER + 缓冲有限 + 信道衰落下 PLR 不可能严格 0**。我们的 0% 是仿真假设下的极限结论；真实部署预期 PLR ≈ 0.01–0.05%，仍**比基线方案低 10–100 倍**。论文应明确写"在理想链路假设下达到 PLR = 0"。

### Q40. MPTCP baseline 是标准 Linux 还是优化版？

是**实现了核心迟滞机制的简化版本**，不是 Linux MPTCP 源码。准确说应称"MPTCP-inspired heuristic"。**辩护**：本工作目的是对比"地面多路径迟滞思想"在卫星的表现，不是 benchmark Linux MPTCP；不同优化版本的 MPTCP 在卫星上都有类似的"迟滞阈值难调"问题。

### Q41. 100% 满载场景表现？

未仿真。本工作业务负载平均 ~50%（300 pps / 网关 2000 pps 能力）。100% 满载下 `_service_time → 1/0` 会让所有方案 buffer 都炸，对比意义不大。这是已知局限。

### Q42. 馈电链路 (Feeder Link) 切换能复用吗？

理论上能，因为 StaticContext / DynamicContext 模型与链路类型无关。**实际差异**：馈电链路有地面接力，可见窗 = 卫星过境时长（5–15 分钟），更长；上下文也更大（含与地面应用的会话状态）。需要做的扩展：把 dynamic 上下文模型扩到地面侧。

### Q43. 非合作目标星座（不同运营商）能用吗？

**当前不能**——本机制假设统一的迁移协议栈。非合作目标需要先建立跨运营商的标准协议（类似 BGP for satellites）。这是产业层而非技术层问题。

### Q44. 激光 ISL + 云层遮挡？

激光 ISL 不受云层影响（在轨道高度云层之上）。但若是星地激光链路（馈电），云层确实会引起短时不可见。本工作 ISL 链路模型可直接适用激光 ISL（带宽改 10–100 Gbps，距离上限改 5000+ km），算法逻辑不变。

### Q45. 算法能耗 vs 节省的传输能耗？

简单算账：每次切换避免 ~15 万分片重传 × 256 B × 5 W·s/MB ≈ 200 J 传输能耗。算法本身（IL 推理 + Lyapunov）每秒 ≈ 0.05 J，30 分钟 = 90 J。**净收益 110 J/切换**，能耗划算。
