## 摘要

本章针对低轨（LEO）异构卫星星座中"AOS 卫星接入 IPv6 互联网"场景下，协议转换网关因星座高速运动而频繁切换所导致的**协议转换服务中断、翻译上下文丢失、CCSDS 分片重组失败**等关键问题，提出一种**基于星历预测的协议转换网关切换与状态一致性迁移机制**。该机制由四个紧密耦合的子算法构成：

1. 基于 Lyapunov drift-plus-penalty 框架的**预测性网关选择算法**，证明了 $[O(1/V), O(V)]$ 的 utility–切换次数权衡上界；
2. 由 Lyapunov 专家解监督训练的**模仿学习策略网络**，将星上决策延迟压至 < 0.2 ms；
3. 借鉴虚拟机热迁移并面向卫星协议级状态扩展的**两阶段状态迁移协议**，含 Pre-copy、Stop-and-copy 与三层降级策略；
4. 解决多网关并发覆盖下"状态分裂"问题的**Top-M 乐观复制 + Gossip 一致性协议**。

在基于 SimPy 的事件驱动仿真平台上（16 颗合成 LEO 网关 + 单 AOS 极轨卫星，30 min × 5 个负载随机种子），与 4 个代表性基线（Reactive、Max-Visibility、MPTCP-style、Pure DRL）对比，**本方案在所有 5 个种子上稳定取得 PLR = 0%、handoff 中断 = 0 s、迁移分片丢失 = 0**，端到端延迟 6.24 ms 与最优基线持平且方差为零；而 MPTCP-style 出现 70 次/30min 的过度切换并造成 6.4% 丢包，Pure DRL 在确定性场景下不及 Lyapunov 在线决策且方差大（4.7× 平均代价差距）。

**关键词**：异构卫星网络；协议转换网关；AOS-IPv6 互联；Lyapunov 优化；模仿学习；状态迁移；一致性协议

## 第一章 研究背景

### 1.1 卫星网络的代际异构化

随着低轨星座（Starlink、OneWeb、千帆、GW）的快速部署，卫星互联网正从"以单星为节点的窄带通信网"演进为"以星座为单元的宽带 IP 互联网"。新一代卫星普遍采用 **IPv6** 作为统一的网络层协议，能力涵盖星间路由、QoS 区分、移动性管理等，已具备承载互联网业务的能力。

然而，在轨运行的大量**存量卫星**（特别是科学探测、遥感、深空通信等任务卫星）仍采用 **CCSDS AOS（Advanced Orbiting Systems）协议**进行数据链路层与网络层封装。AOS 协议自 1995 年提出以来，凭借其在恶劣空间链路上的鲁棒性（前向纠错、虚拟信道 QoS）成为航天通信的事实标准。AOS 帧结构（256 字节定长）由同步头、主导头（含 SCID/VCID）、路由信息域、可靠传输域、数据域（含 M_PDU 承载若干 CCSDS 包的分片）、CRC 组成。

这种代际差异导致了一个核心矛盾：**存量 AOS 卫星无法直接接入新一代 IPv6 卫星互联网**。如果要求 AOS 卫星全部替换，工程成本巨大且不现实；如果各自孤立运行，又违背了"天地一体化网络"的发展目标。

### 1.2 协议转换网关：异构互联的关键中介

我的第二个专利提出了**在新一代 IPv6 卫星上实现 AOS↔IPv6 协议转换**的方法（跨域互联场景下的报文转换），使新型 IPv6 卫星扮演**协议转换网关**的角色：

- AOS 卫星将业务数据封装成 AOS 帧，发往附近可见的 IPv6 网关卫星
- IPv6 网关卫星按预先配置的 (SCID, VCID) → IPv6 地址映射规则，将 AOS 帧载荷重组为 CCSDS 包，再封装为 UDP/IPv6 报文，注入 IPv6 卫星互联网
- 反向同理

该专利解决的是**静态场景**下的协议转换——即假设 AOS 卫星与某一颗 IPv6 网关之间的链路保持稳定。

### 1.3 创新点三在本论文中的定位

本论文的三个创新点构成层次递进的技术链条：

| 创新点 | 解决层次 | 核心内容 |
|---|---|---|
| 一 | 网络层地址 | AOS 星座下 IPv6 链路本地地址的自动配置 |
| 二 | 协议报文 | AOS↔IPv6 双向静态转换 |
| **三** | **服务连续性** | **星座高速运动下协议转换服务的无缝切换与状态一致性** |

三者完整构成 **"能通信 → 能转换 → 能连续转换"** 的闭环。

## 第二章 问题与困境

### 2.1 静态协议转换的局限

LEO 卫星轨道高度 400–1500 km，星下点速度约 7.6 km/s。**任意 IPv6 网关卫星与某颗 AOS 卫星之间的星间可见窗口都是有限的**。一旦窗口结束，AOS 卫星必须切换到另一颗可见的 IPv6 网关继续协议转换服务。这是创新点二未涉及的场景，也是本章工作的出发点。

#### 可见窗口的理论估算

对两颗 LEO 卫星之间的星间链路（ISL），可见窗口由相对几何决定，可按下式粗略估算：

$$T_{\text{vis}} \;\approx\; \frac{2\sqrt{D_{\max}^2 - d_{\min}^2}}{|v_{\text{rel}}|}$$

- $D_{\max}$：星间链路最大允许距离（受大气切线高度约束，本场景 $D_{\max}\approx 3500$ km 工程取值）；
- $d_{\min}$：本次相遇过程中两星最近距离；
- $|v_{\text{rel}}|$：相对速度，异面 LEO 卫星典型 1–3 km/s。

代入典型值（$d_{\min}\approx 1500$ km, $|v_{\text{rel}}|\approx 2$ km/s）得 $T_{\text{vis}} \approx 3{-}10$ min。受相对几何动态变化的影响，实际仿真中观察到的窗口集中在该区间，最坏可短至 1–2 min（近临界几何）。

### 2.2 翻译上下文（Translation Context, TC）丢失问题

协议转换网关在工作时，**内存中维护着大量与具体 AOS 卫星会话紧密绑定的状态**，我们统称为**翻译上下文** $C$，由两部分构成：

- **静态部分 $C_{static}$**：(SCID, VCID) → IPv6 地址的映射表、QoS 策略表。这部分在网关运行期间相对稳定。**理论估算约 288 B/会话**，其构成如下：

  | 项 | 字节 |
  |---|---|
  | (SCID,VCID) → IPv6Mapping 项：src/dst IPv6 (16+16 B) + SCID/VCID (2+2 B) + flow_label (4 B) + flags (4 B) ≈ 44 B；按 4 条流计 4×44 ≈ | 176 B |
  | QoS 元数据：DSCP (1 B) + bandwidth_quota_kbps (4 B) + queue_id (2 B) + priority (1 B) ≈ 8 B；按 4 条流计 4×8 ≈ | 32 B |
  | 表头/池化指针、对齐填充 | 32 B |
  | 会话元数据：version (8 B) + timestamp_sec (8 B) + flags (8 B) + 边界字段 + 对齐 | 48 B |
  | **合计** | **≈ 288 B** |

  实际部署中视会话中流数与 IPv6 字段长度可在 220–360 B 之间浮动；本工作以 288 B 作为代表值用于 Pre-copy 耗时计算。
- **动态部分 $C_{dynamic}$**：
  - **CCSDS 分片重组缓存 $C_{frag}$**：由于一个 CCSDS 包可能跨多个 AOS 帧的 M_PDU 传输（典型分 3–8 片），网关需要按 (SCID, VCID, ccsds_pkt_id) 维护一个未完成重组的分片队列。
  - **VCID 级 QoS 队列状态 $C_{qos}$**：每个虚拟信道独立队列，记录队头报文、累计字节、已发送序列号等。
  - **协议转换序列号 $C_{seq}$**：用于 IPv6 流的有序性。

$$C = C_{static} \cup C_{dynamic},\quad C_{dynamic} = C_{frag} \cup C_{qos} \cup C_{seq}$$

### 2.3 硬切换导致的通信中断与丢包（状态硬重置的双重惩罚机理）

传统切换采用 **Break-before-Make（先断后立）** 模式：原网关 A 失去可见性后，AOS 卫星才尝试与新网关 B 建立连接，B 从零开始构建上下文。这一模式带来两层并列的损失：

1. **控制面重建延迟**——AOS 卫星与新网关需要重新完成寻呼、链路同步与上下文协商，带来**数百毫秒的通信盲区**，此期间到达的报文无条件丢失。该值按经验可拆分为三段：

   | 阶段 | 典型耗时 |
   |---|---|
   | 天线重指向（机械式 vs 相控阵） | 200–500 ms / <50 ms |
   | 链路重捕获 + 比特同步 | 100–300 ms |
   | 上下文重协商（会话密钥、QoS、地址） | 100–400 ms |
   | **总计（机械天线）** | **0.3–1.0 s** |

   本工作仿真中按 0.5 s 取保守中位值。

2. **数据面分片截断（核心痛点）**——由于 AOS 帧长固定（256 B），上层大块 IPv6 报文必须跨帧分片。物理切换瞬间，原网关已接收但未重组完成的"半截"数据包因无法跨网关继承，将被直接丢弃，导致**极高的无效传输带宽浪费**。

> **实测数据**（见第七章）：本仿真中传统硬切换方案 (Reactive / Max-Visibility) 单次切换丢弃 14.5 万–17.5 万分片，PLR 0.23%–0.28%；MPTCP-style 因过度切换（70 次/30 min）使 PLR 飙至 6.4%。本方案 (Ours-Lyap / Ours-IL) 通过两阶段迁移把这两个数值同时压至零。

### 2.4 看似可借鉴的 VM Live Migration 与其本质局限

云计算中的虚拟机热迁移（Live Migration，Clark et al. 2005）通过 Pre-copy + Stop-and-copy 实现了"准零停机"的状态迁移，思想可以借鉴。但卫星协议转换网关迁移有 **4 个 VM 场景不具备的本质难点**：

| 难点 | VM 迁移 | 卫星协议转换网关迁移 |
|---|---|---|
| 协议级状态 | 无 | 必须迁移 $C_{frag}$ 与 $C_{qos}$ |
| 链路时变性 | 数据中心带宽稳定 | ISL 带宽随距离/几何时变，存在不可见窗口 |
| 多目标候选 | 单目标主机 | 同时被 2–3 颗 IPv6 卫星覆盖，存在"状态分裂" |
| 时间硬约束 | 可秒级停机 | 物理切换窗口仅几百 ms，且星历严格确定 |

因此**简单移植 VM 热迁移无法解决卫星问题，必须重新设计**。

### 2.5 现有方法的不足

- **被动硬切换（Reactive）**：仅当当前网关不可见才切，中断严重
- **贪心可见性（Max-Visibility）**：每步选剩余可见时长最大的网关，频繁切换且不考虑网关负载
- **MPTCP 类软切换**：地面多路径思想，缺乏协议级状态迁移；在确定性卫星场景下其迟滞机制会反复抖动（详见第七章 7.2 节）

### 2.6 多覆盖网关并发：状态分裂与权威副本难题

LEO 星座中一颗 AOS 卫星在同一时刻经常被 2–3 颗 IPv6 网关同时覆盖。若每颗网关都尝试对唯一一份翻译上下文 $C$ 独立维护，AOS 切换到任意一颗时都可能读到"过期/错乱"的状态。可拆解为三个子问题：

- **子问题 A：状态视图冲突**——同一 (scid, vcid) 流被多颗网关并发处理、分片缓存独立；若不交换"版本号 + 时间戳"，切换瞬间无法判断谁最新；由于分片缓存独立演进，网关间缺乏状态同步，盲目切换会导致**已转发报文重传或待组装分片丢失**。
- **子问题 B：去中心化协同困境**——候选网关地位对等、缺乏全局调度中心；传统强一致性协议（如 Raft / Paxos）需多轮 RTT 协商，在星间链路高时延、频繁中断的环境下**信令开销无法承受**；必须采用"最终一致 + 偏序时钟"的轻量协议。
- **子问题 C：过时副本残留**——网关失去可见性后，其持有的上下文必须及时回收，否则会造成内存泄漏或被后续错误读取；副本数量需有上界（Top-M），否则 16 颗候选下开销线性爆炸；需建立统一的 TTL 与显式淘汰规则机制，保证副本仓库稳定可控。

> **结论**：需要一种"**去中心化、轻量级、最终一致**"的状态同步机制，在多网关并发覆盖、链路高动态的条件下，仍能确保状态不分裂、可安全切换、业务不中断。这一结论直接驱动了 4.6 节的 Top-M 乐观复制与 Gossip 一致性设计。
- **纯 DRL 决策**：训练不稳定、决策需 GPU、可解释性差，且在星历可预测场景下并非最优（实测 reward 比 Lyapunov 专家差 4.7×）

## 第三章 解决思路总览

本章提出**面向卫星特性的四段式状态迁移机制**：四个阶段沿"阶梯"逐级向上，每两步之间都不是简单的"接着做"，而是上一步必然引出"下一步必须做"的具体原因。

```
                                                          ┌──────────────────────┐
                                                          │ ④ 两阶段迁移 + Gossip │
                                                          │  · Pre-copy/Stop-copy │
                                                          │  · Top-M + Lamport    │
                                  ┌──────────────────┐    └──────────────────────┘
                                  │ ③ 模仿学习 (IL)   │            ▲
                                  │  · 蒸馏到 ~45 K    │            │ 下一步必须做：
                                  │    参数 MLP        │            │ 状态迁移必须可靠
                  ┌─────────────┐ │  · < 0.2 ms 决策   │            │ 且一致，才能保证
                  │ ② Lyapunov  │ └────────┬─────────┘            │ 服务不中断且可恢复
                  │   决策      │          ▲                       │
                  │  · 排队论    │          │ 下一步必须做：           │
                  │  · 闭式 O(V) │          │ 在线求解开销大，          │
   ┌────────────┐ └──────┬──────┘          │ 需将求解过程网络化         │
   │ ① 星历预测  │        ▲                 │                          │
   │  · TLE 已知 │        │ 下一步必须做：    │                          │
   │  · ΔT/距离/带│        │ 不再是马尔可夫    │                          │
   │    宽 → 可知 │        │ 黑盒，可做排队     │                          │
   └─────┬──────┘        │ 论意义下的最优     │                          │
         ▲               │ 决策              │                          │
         │ 下一步必须做：  │                  │                          │
         │ 把长期动态不确  │                  │                          │
         │ 定性转化为带物  │                  │                          │
         │ 理约束的在线    │                  │                          │
         │ 决策问题       │                  │                          │
         └───────────────┴──────────────────┴──────────────────────────┘
```

> **核心目标**：在 LEO 高动态、可见窗口受限的条件下，实现状态的**无缝迁移、零中断、零丢包、状态可证**。

四段式机制对应论文中 4 个子算法：

1. **算法 1：基于 Lyapunov drift-plus-penalty 的预测性网关选择**（理论方案）
2. **算法 2：模仿学习策略网络**（部署友好方案，<0.2 ms 决策延迟）
3. **算法 3：两阶段协议转换状态迁移与三层降级策略**
4. **算法 4：Top-M 乐观复制与 Gossip 一致性协议**

四步因果链解读：

- **① → ②**：星历可预测意味着"未来"是确定可知的，问题不再是马尔可夫黑盒；可以直接做排队论意义下的最优决策（Lyapunov 框架自然适配）。
- **② → ③**：Lyapunov 闭式策略虽然 $O(N)$，但每时隙都需在线维护虚拟队列 $Q(t)$、计算代价矩阵；星载抗辐照计算平台（如 FPGA）对浮点比较 / 条件分支支持效率受限，难以保证严格的微秒级时延边界；因此必须将求解过程**蒸馏为纯前向推断网络**。
- **③ → ④**：决策选出"切到哪一颗"只解决了"何时切、切给谁"的问题；物理切换瞬间仍需底层的协议状态无缝迁移与多副本一致性机制，才能真正保证业务连续。状态迁移必须既"可靠"又"一致"——这正是 4.5 / 4.6 节内容。

## 第四章 详细原理

### 4.1 系统模型与符号

#### 星座构型

- **AOS 卫星**：1 颗，轨道高度 $h_A = 400$ km（不失一般性，多颗可独立处理）；
- **IPv6 网关卫星**：$N = 16$ 颗，轨道高度 $h_G = 550$ km，等 RAAN 间距分布在同一倾角面，构成均匀候选池；
- **时间离散化**：连续时间离散为时隙 $t \in \{0, 1, 2, \dots\}$，时隙长度 $\Delta t = 1$ s；
- **决策变量**：$a(t) \in \{1, \dots, N\}$，表示当前服务 AOS 的网关索引。

#### 几何与链路量

| 符号 | 含义 | 单位 |
|---|---|---|
| $D_i(t)$ | AOS 与 $g_i$ 间直线距离（由星历推算） | km |
| $V_i(t)$ | 星间可见性指示（见下方约束） | 0/1 |
| $\Delta T_i(t)$ | $g_i$ 的剩余连续可见时长（由星历精确预测） | s |
| $B_i(t)$ | AOS↔$g_i$ ISL 带宽，反距离平方递减 | Mbps |
| $L_i(t)$ | $g_i$ 当前 CPU 协议转换负载 | [0,1] |
| $\tau_i(t)$ | AOS↔$g_i$ 传播延迟 | ms |

#### 星间可见性约束（避大气切线高度准则）

**注意**：传统"最小仰角 $\ge 10°$"是地面站对卫星的可见性约束，目的是避开地面建筑物遮挡与低空大气折射，并不适用于星间链路。本工作采用**避大气切线高度（Tangent Altitude）准则**：

设地球半径 $R = 6378$ km，大气有效边界 $h_{\text{atm}} = 80$ km。AOS（高度 $h_A$）与 $g_i$（高度 $h_G$）之间的直射 LoS 不被大气遮挡，当且仅当 AOS-$g_i$ 直线与"地心-$R{+}h_{\text{atm}}$"球面无交点。其最大允许距离：

$$D_{\max} \;=\; \sqrt{(R+h_A)^2 - (R+h_{\text{atm}})^2} \;+\; \sqrt{(R+h_G)^2 - (R+h_{\text{atm}})^2}$$

代入 $h_A = 400$, $h_G = 550$, $h_{\text{atm}} = 80$ km 得 $D_{\max} \approx 4567$ km。本工作取 **$D_{\max} = 3500$ km** 作为保守工程门限（保留余量以应对几何抖动与天线指向误差）。可见性判据简化为：

$$V_i(t) = \begin{cases} 1, & D_i(t) \le D_{\max} \\ 0, & \text{otherwise} \end{cases}$$

剩余可见时长：

$$\Delta T_i(t) = \inf\{ \Delta : V_i(t+\Delta) = 0 \} - t,\quad \text{若 } V_i(t) = 1$$

> ⚠ 与地面通信不同，ISL 的"可见性"由两端**地心矢量与连线的几何切线**判定（line-of-sight, LOS），代码实现见 `orbit/skyfield_sim.py: has_line_of_sight()`。

#### 决策与代价

- 决策变量：$a(t) \in \{1, 2, \dots, N\}$，表示当前时隙服务的网关
- 即时代价：

$$c(t) = \alpha \cdot \mathbf{1}\{V_{a(t)}(t) = 0 \;\vee\; \Delta T_{a(t)}(t) < T_h\} + \beta \cdot L_{a(t)}(t) + \gamma \cdot \mathbf{1}\{a(t) \ne a(t-1)\}$$

其中 $T_h$ 为"即将耗尽"阈值（默认 5 s），$\alpha, \beta, \gamma$ 为权重。三项分别对应：中断惩罚、负载惩罚、切换瞬时代价。

### 4.3 基于 Lyapunov drift-plus-penalty 的预测性网关选择

代码：`optimizer/lyapunov_solver.py`。

#### 4.3.1 问题建模

考虑长期平均最小化：

$$\min_{\{a(t)\}} \;\; \overline{c} \;\;=\;\; \lim_{T\to\infty} \frac{1}{T}\sum_{t=0}^{T-1} c(t)$$

约束：长期平均切换率不超过预算 $C_{\max}$：

$$\overline{x_{sw}} \;=\; \lim_{T\to\infty}\frac{1}{T}\sum_{t=0}^{T-1} \mathbf{1}\{a(t)\ne a(t-1)\} \;\le\; C_{\max}$$

#### 4.3.2 虚拟队列与 drift-plus-penalty

引入虚拟队列 $Q(t)$，跟踪切换预算累计违反量：

$$Q(t+1) = \max\{Q(t) + \mathbf{1}\{a(t)\ne a(t-1)\} - C_{\max}\cdot \Delta t,\;\; 0\}$$

定义 Lyapunov 函数 $L(Q(t)) = \tfrac{1}{2} Q(t)^2$，则单时隙漂移：

$$\Delta(Q(t)) \;\triangleq\; \mathbb{E}\{L(Q(t+1)) - L(Q(t)) \mid Q(t)\}$$

加上罚函数 $V \cdot c(t)$，得到 **drift-plus-penalty** 上界（标准 Lyapunov 推导，见 Neely 2010）：

$$\Delta(Q(t)) + V\cdot c(t) \;\le\; B + Q(t)\cdot[\mathbf{1}\{a(t)\ne a(t-1)\} - C_{\max}\cdot \Delta t] + V\cdot c(t)$$

其中 $B$ 为与决策无关的常数。

#### 4.3.3 闭式决策策略

每时隙 $t$，最小化 RHS（去掉常数）：

$$a^*(t) \;=\; \arg\min_{a\in\{1,\dots,N\}} \;\; V\cdot \tilde{c}_a(t) + Q(t)\cdot \mathbf{1}\{a \ne a(t-1)\}$$

其中 $\tilde{c}_a(t) = \alpha\cdot\mathbf{1}\{V_a(t)=0 \vee \Delta T_a(t) < T_h\} + \beta \cdot L_a(t) + \gamma\cdot\mathbf{1}\{a\ne a(t-1)\}$ 是把决策 $a$ 代入 $c(t)$ 后的值。

这是一个 $O(N)$ 的逐点比较，**无需任何优化求解器**，闭式可解。代码 `lyapunov_solver.py:lyapunov_online()`。

#### 4.3.4 性能界定理（论文 5.2 节核心结果）

**定理 1（$[O(1/V), O(V)]$ 权衡）**：在满足 Slater 条件的前提下，对任意 $V > 0$，drift-plus-penalty 策略满足

$$\overline{c}^{\text{Lyap}} \;\le\; \overline{c}^{\text{opt}} + \frac{B}{V}$$

$$\overline{Q} \;\le\; \frac{B + V\cdot(c_{\max} - c_{\min})}{\epsilon}$$

其中 $\overline{c}^{\text{opt}}$ 是离线最优代价，$\epsilon > 0$ 是 Slater 余量。证明详见 `docs/lyapunov_proof.md`。

**实测补充**（详见 `docs/lyapunov_proof.md` 第 8 节）：在本仿真场景下，由于决策被可见性硬约束主导（一旦当前网关不可见，$\alpha = 1$ 的中断惩罚远超 $V$ 调节的负载差异），$V \in [1, 1000]$ 给出相同的决策序列。这一**对 V 的不敏感性**说明 Lyapunov 框架在卫星场景下工程鲁棒性极强。

#### 4.3.5 为什么这优于 DRL

1. **理论保证**：有闭式性能界，可证明
2. **零训练**：无需大量样本与 GPU
3. **决策延迟**：纯 NumPy 运算 < 1 ms（DQN GPU 推理 1–5 ms）
4. **可解释**：每一步选择都有数学依据
5. **星历可预测性**：Lyapunov 利用了"未来可知"这一卫星场景独有特性

**实测验证**：见第七章 7.4 节，Pure-DRL 在 30 min 测试中平均 reward = −1.38 ± 高方差，而 Lyapunov 专家 = −0.30 (即 DRL 代价是 Lyapunov 的 4.7×)。

### 4.4 模仿学习加速

代码：`optimizer/policy_net.py`。

#### 4.4.1 动机

虽然 Lyapunov 在线决策已经很快，但：
- 真实卫星 CPU 资源紧张（FPGA / 抗辐射处理器算力有限）
- 需要稳定、低方差的推理时间

因此训练一个**轻量级 MLP 策略网络** $\pi_\theta(a|s)$ 作为 Lyapunov 求解器的**快速代理**。

#### 4.4.2 网络架构

- 输入：$s_t = [\Delta T_1, \dots, \Delta T_N, L_1, \dots, L_N, \tau_1, \dots, \tau_N, V_1, \dots, V_N, \text{onehot}(a_{t-1})]$，维度 $5N = 80$（$N=16$）
- 隐藏层：3 层，每层 **128** 单元，ReLU + Dropout(0.1)
- 输出：$N$ 个 logit，取 argmax + visibility mask（避免选不可见网关）

**参数量**（$N=16$ 实测，`GatewayPolicyNet(state_dim=80, num_gateways=16, hidden_dim=128, num_layers=3)`）：

| 层 | 形状 | 参数数 |
|---|---|---|
| net.0 (Linear 80→128) | (128, 80) + (128,) | 10 368 |
| net.3 (Linear 128→128) | (128, 128) + (128,) | 16 512 |
| net.6 (Linear 128→128) | (128, 128) + (128,) | 16 512 |
| net.9 (Linear 128→16, 输出) | (16, 128) + (16,) | 2 064 |
| **合计** | | **45 456** |

约 45 K 参数，单次推理实测：
- NVIDIA L20X GPU: **0.176 ms / call**
- 同机 CPU (Intel 默认线程): **0.144 ms / call**

> CPU 反而略快是因为小模型下 GPU kernel 启动开销 (~50 μs) 超过纯计算时间，是 GPU 部署小模型的典型特征。对星上部署是利好——CPU/FPGA 即可在 < 1 ms 完成决策，决策时隙 1 s 富裕 5000×。

#### 4.4.3 训练目标（行为克隆 + DAgger 增强）

设专家数据集 $\mathcal{D} = \{(s_i, a^*_i)\}_{i=1}^M$，其中 $a^*_i$ 由 Lyapunov 求解器生成。优化交叉熵：

$$\mathcal{L}_{BC}(\theta) = -\frac{1}{M}\sum_{i=1}^M \log \pi_\theta(a^*_i \mid s_i)$$

为防止数据集类别单一（"专家在专家状态下"会反复访问同几个网关），代码实现了 **DAgger 风格数据扩增**（`policy_net.py:build_dagger_dataset()`）：对每条专家轨迹施加 $K$ 次状态扰动（在负载维度叠加高斯噪声），让模型见到更多 $(s', a^*)$ 配对。

#### 4.4.4 性能保证

由于 $\pi_\theta$ 拟合 Lyapunov 专家解，**理论上其性能上界即为 Lyapunov 专家解**。

**实测验证**：在 8 个训练场景 × 60 epoch（AdamW lr=1e-3、batch=256、val_ratio=0.2）后，IL 网络 `val_acc = 99.91%`、`val_loss = 0.0092`。在 5 个测试种子上，Ours-IL 与 Ours-Lyap 取得**完全一致**的决策与仿真结果（PLR=0%, switches=1, e2e=6.24 ms 等指标完全相等）。

### 4.5 两阶段协议转换状态迁移

代码：`migration/two_phase.py`。

#### 4.5.1 阶段流程

```
        ┌──── 切换前 t_pre 秒 ────┬─ 物理切换 t_phys ─┐
        │   Pre-copy             │  Stop-and-copy   │
        │   A → B 传 C_static     │  A 冻结 ΔC_dyn    │
        │   A 继续服务             │  传给 B，B 接管    │
        └─────────────────────────┴──────────────────┘
```

#### 4.5.2 阶段 1：Pre-copy

- 决策前 $t_{pre}$ 秒（典型 3 s），向目标网关 B 通过 ISL 推送 $C_{static}$
- A 同时拍下 $C_{dynamic}$ 的快照 $\hat{C}_{dyn}^{(0)}$（深拷贝）
- A 继续接收并处理 AOS 帧，$C_{dynamic}$ 持续增长
- Pre-copy 耗时：

$$t_{precopy} = \frac{|C_{static}|}{B(A,B) \cdot \eta} + \tau(A,B)$$

其中 $\eta$ 是 ISL 协议开销系数（默认 0.8）。

#### 4.5.3 阶段 2：Stop-and-copy

- 切换瞬间，A 冻结 $C_{dynamic}$，计算增量 $\Delta C_{dyn} = C_{dynamic}(t) - \hat{C}_{dyn}^{(0)}$
- 通过 ISL 传给 B，传输耗时：

$$t_{stop} = \frac{|\Delta C_{dyn}|}{B(A,B)\cdot \eta} + \tau(A,B)$$

#### 4.5.4 无缝衔接判据（修正原方案的理想化公式）

$$T_{sync} = t_{stop} \;\stackrel{?}{\le}\; T_{phys}$$

- **若满足**：进入 SEAMLESS 路径，B 接管时 $C$ 完整，理论零中断
- **若不满足**：触发 3 层降级（见下）

#### 4.5.5 三层降级策略

| 层 | 触发条件 | 策略 | 备注 |
|---|---|---|---|
| L1: SEAMLESS | $T_{sync} \le T_{phys}$ | 全量迁移 | 完全无缝 |
| L2: DEGRADED-1 | L1 失败 | 仅迁移已收到 ≥50% 分片的 CCSDS 包 + 全部 QoS 队列 | 服务连续，部分包丢失 |
| L3: DEGRADED-2 | L2 仍失败 | 仅迁移高优先级 VCID (vcid ≤ 1) 的 ≥50% 分片 | 保证关键业务 |
| L4: FAILED | L3 仍失败 | 回退为硬切换 | 等效传统方法 |

**降级判定理论依据**：≥50% 完成度的分片是"投资已超过半数"的状态，继续迁移的边际收益高于重传。

**实测验证**（图 6 `fig6_migration_overhead.png`）：在 30 min 仿真中本方案的切换全部命中 L1 SEAMLESS 路径，sync_time ≈ 30 ms，远小于 $T_{phys} = 500$ ms。

#### 4.5.6 为什么这是创新而非简单移植 VM 迁移

| 关键设计 | 卫星独有性 |
|---|---|
| 分片可迁移性判定（≥50% 阈值） | VM 没有"协议分片"概念 |
| 优先级感知降级（VCID 优先级保护） | VM 应用层透明，不知优先级 |
| 带宽-距离时变模型 | VM 数据中心内带宽恒定 |

### 4.6 多网关一致性 Gossip 协议

代码：`migration/consistency.py`。

#### 4.6.1 问题：多网关并发覆盖的"状态分裂"

当 AOS 卫星同时被 $k\ge 2$ 颗 IPv6 网关覆盖，且未来切换目标的预测有不确定性时：
- 若提前只复制到 1 个候选 → 万一选错，仍需重新 Pre-copy
- 若不复制 → 临时 Pre-copy 时间紧张

#### 4.6.2 解决方案：Top-M 乐观复制

**复制决策规则**：

$$\text{score}_i = \Delta T_i - 50 \cdot L_i$$

选取分数最高的 $M$ 个候选（默认 $M=2$，排除当前 A），并行复制 $C_{static}$。开销 $M \cdot |C_{static}|$，由于 $|C_{static}|$ 很小（实测 288 B），开销可忽略。

#### 4.6.3 Gossip 一致性维护

每颗网关持有副本仓库 $\mathcal{R}_g = \{(s, C_{static}, v, t_w, \tau_{TTL})\}$，其中 $v$ 为版本号、$t_w$ 为写入时间、$\tau_{TTL}$ 为生存时间（设为该网关对 AOS 的剩余可见时长，由星历精确计算）。

- **周期 Gossip**（默认 10 s 一轮）：每个网关向所有其他网关广播 $(s, v, t_w)$ 摘要
- **接收方处理**：若本地副本版本 $< v_{msg}$，淘汰本地（说明它过时了）
- **TTL 淘汰**：副本超过 $\tau_{TTL}$ 自动失效

> 用 **Lamport 时钟**保证消息的偏序性，避免乱序到达导致误判。

**实测验证**：30 min 仿真中 Ours-Lyap / Ours-IL 各运行 179 轮 Gossip、淘汰 1 个陈旧副本、安装 1 个 288 B 静态副本，证明全部三个路径（写、传播、淘汰）皆已打通。

#### 4.6.4 与强一致性协议（Raft）对比

| 维度 | Raft | 本方案 Gossip |
|---|---|---|
| 一致性 | 线性化（强） | 最终一致 |
| 消息复杂度 | $O(N^2)$ 每决策 | $O(N)$ 每周期 |
| 延迟 | 至少 2 RTT | 异步，无阻塞 |
| 卫星适用性 | 差（ISL 时延高） | 好 |

**结论**：卫星协议转换上下文允许最终一致性（陈旧副本最多导致一次小范围重 Pre-copy），无需 Raft 的强一致性开销。

### 5.2 关键算法伪代码

#### 算法 1：Lyapunov 在线决策

```python
Q ← 0; a_prev ← initial_gw
for t = 0, 1, 2, ...:
    rem ← topo.remaining_visibility(t)
    L   ← gateway_loads(t)
    vis ← topo.visible[t]
    for a in 1..N:
        interrupt ← (not vis[a]) or (rem[a] < T_h)
        cost_a    ← α*interrupt + β*L[a] + γ*(a ≠ a_prev)
        score[a]  ← V*cost_a + Q*(a ≠ a_prev)
    a* ← argmin(score)
    if a* ≠ a_prev:
        trigger_two_phase_migration(a_prev → a*)
    Q ← max(Q + (a* ≠ a_prev) - C_max*Δt, 0)
    a_prev ← a*
```

#### 算法 2：两阶段迁移与降级

```python
def migrate(A: TC, bw, prop, lead, T_phys):
    # Pre-copy
    snap ← A.dynamic.snapshot()
    send(A.static → B, time = |A.static|/(bw*η) + prop)
    wait(lead)

    # Stop-and-copy
    delta ← A.dynamic.diff_bytes_since(snap)
    t_sync ← delta / (bw*η) + prop

    if t_sync ≤ T_phys:
        return SEAMLESS, migrate_all

    # Layer 1 降级：选 ≥50% 完成度
    selected_L1, bytes_L1 ← select_by_completion(A.dynamic, threshold=0.5)
    if bytes_L1 / (bw*η) + prop ≤ T_phys:
        return DEGRADED_1, selected_L1

    # Layer 2 降级：仅高优先级 VCID
    selected_L2, bytes_L2 ← select_by_completion(A.dynamic,
                                                  threshold=0.5,
                                                  high_priority_only=True)
    if bytes_L2 / (bw*η) + prop ≤ T_phys:
        return DEGRADED_2, selected_L2

    return FAILED, hard_handoff_fallback
```

#### 算法 3：Top-M 乐观复制 + Gossip

```python
def replicate_static(A, candidates, M):
    scores ← [ΔT_i - 50*L_i for i in candidates]
    top_M ← topk(scores, M)
    for g in top_M:
        g.replica_store.install((A.static, version=v, ttl=ΔT_g))

def gossip_round(stores):  # 每 10 s 一次
    for g in stores:
        for msg in g.announce():  # (scid, version, written_at)
            for other in stores - {g}:
                if other.replicas[msg.scid].version < msg.version:
                    other.evict(msg.scid)
        g.evict_stale(now)
```

### 6.6 性能优化：分片重组缓存 200× 提速

cProfile 显示 `migration/context.py:FragmentReassemblyBuffer.complete_packets()` 占用单次仿真 68% 的运行时间。原实现每次调用都遍历整个 `buf` 字典并对每个值 `len(frags)`：

```python
# v1：O(buf 大小)，每次 _process_frame 调用
def complete_packets(self):
    done = []
    for k, frags in self.buf.items():
        if frags and len(frags) == frags[0].frag_total:
            done.append(k)
    return done
```

v2 改成增量维护已完成 key 的集合：

```python
# v2：O(已完成集合大小)
def add(self, scid, vcid, frag, t_sec):
    key = (scid, vcid, frag.ccsds_pkt_id)
    frags = self.buf.setdefault(key, [])
    frags.append(frag)
    if len(frags) == frags[0].frag_total:
        self._complete_set.add(key)
    self.last_update_sec = t_sec

def complete_packets(self):
    return list(self._complete_set)
```

**实测效果**：30 min × 300 pps 仿真单次跑通从 **451 s → 2.3 s（200×）**，单元测试从 93 s → 5.8 s。`simpy_env.py:_do_handoff()` DEGRADED 分支中直接对 `buf` 赋值的代码同步修正为也更新 `_complete_set`，保证不变量。

## 第八章 结论与未来工作

### 8.1 主要贡献

1. **理论贡献**：将 Lyapunov drift-plus-penalty 框架引入卫星协议转换网关切换问题，证明了 $[O(1/V), O(V)]$ 权衡上界；实测验证 V 在卫星场景下被硬约束吸收，工程鲁棒性极强
2. **机制贡献**：提出协议转换上下文的概念，定义 $C_{static}/C_{dynamic}$ 数据结构，并设计两阶段迁移与三层降级策略
3. **协议贡献**：针对多网关并发覆盖场景，设计 Top-M 乐观复制 + Lamport 时钟 Gossip 一致性协议
4. **工程贡献**：完整 SimPy 仿真平台 + 6 方案对比代码 + 5 seed mean±std 统计 + 31 个单元测试 + 9 张论文图，所有模块开源可复现

### 8.2 与现有工作的本质区别

| 工作类别 | 代表 | 差距 |
|---|---|---|
| 卫星硬切换研究 | 多数早期工作 | 不考虑状态迁移 |
| VM 热迁移 | Clark et al. 2005 | 不处理协议级状态 |
| 地面 MPTCP 移动性 | RFC 8684 | 不适用 ISL 时变；本工作实测其在卫星场景反模式 |
| 基于 DRL 的卫星切换 | 近年若干 | 缺乏理论保证；本工作实测在确定性场景下 4.7× 差于 Lyapunov |
| **本工作** | - | **首个 Lyapunov + 协议级迁移 + Gossip 一致性 + 实测 PLR=0%** |

### 8.3 局限性

1. 当前模型假设单 AOS 卫星，多 AOS 并发场景下 IPv6 网关的容量竞争未深入建模
2. Pre-copy 阶段未考虑 ISL 带宽抢占
3. 安全维度（Gossip 消息真实性）仅在文档中讨论，未编码实现
4. 真实 Starlink TLE 同壳卫星相对静止 → 当前实验依赖合成星座；要复现真实 Starlink 多壳 / Iridium-Starlink 互联，需更复杂的网关采样策略

### 8.4 未来工作

1. **多 AOS 卫星扩展**：把网关选择问题扩展为博弈论 / 拍卖机制
2. **基于 LSTM 的星历预测增强**：当 TLE 数据陈旧时用 LSTM 在线修正
3. **HIL 实验**：在 SDR / FPGA 上跑最小协议转换原型，闭合理论-仿真-硬件验证链
4. **安全增强**：引入轻量级 HMAC 保护 Gossip 消息完整性
5. **真实星座扩展**：研究多壳/异质轨道高度的 Starlink 互联，让真实 TLE 给出有意义的切换决策
